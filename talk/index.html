Recap Mobile Architecture

  • Occasionally connected, infrequent better
  • High Latency, High Throughput, High Variance
  • Responsive, Adaptive, Gracefully Degradable
  • Pervasive: Mobile First, Fluid, Multi/Omni-Screen/Device
  • Out of Control, Multi-Version, Unsecured Access
Out of Control in terms of Fragmentation, Device Manufacturers, App Store Delivery, Upgrade Cycles, Proxy/Network Infrastructure etc.

Recap Mobile Integration

    Access
  • Verb: State synced through commands, minimal data
  • Noun: State and Data transition, minimal commands
    Concurrency
  • Sync: MVC/3-Tier, Form submission, RMI/SOFEA, Snapshot Versioning
  • Async: Android Providers + Intents, iOS Core Data + Queue

=> Noun/Sync a.k.a. Data Synchronization prevalent (e.g. SAP)

  • Hypermedia can be leveraged
  • ROCA application patterns resemble the Web

Hybrid Development

Hybrid HTML 5 is ready

  • Natural choice as problems solved in HTTP
  • Security and caching can be customized in native container
  • Device UI argument gets weaker (e.g. Material Design)
  • Traction to go HTML 5 (SAP, IBM, Titanium, Kony, Xamarin)

Distributed Processing on Resources

Offline First vs. traditional Sync

  • Hybrid changes Integration to more Async (Single Thread
    updates DOM as ViewModel), use of HTML 5 Storage
  • Integration model includes State Transfer
  • Data-binding on MVVM allows fine-granular sync e.g.
    Backbone provides Binding and Event Bus
  • Can be implemented alongside CQRS
Angular goes MVVM, Clean HTML 5, complex structure (Tables etc) discouraged). Links to Microsoft as they were quite early with those patterns and enabled LINQ to work with

Single Page Web Apps

Using the Browser as Engine for RIA

  • Single Page Application pattern includes Routing, Templating
  • Very similar to Hybrid Applications, only Framework native
  • AngularJS provides separation, better Testability
  • Dependencies tricky, look at WebJars and Yeoman/Bower/RequireJS
  • Use Progressive Enhancement, Server Rendering, then Changes
  • E.g. LazoJS and Rendr(uses Backbone)

Future of Interfaces

Omni/Multi/Cross-Channel approaches

  • Multiple Screens/Devices at the same time
  • Network session decoupled from User Session
  • Multiscreen cannot be solved efficiently with Databinding
  • Hybrid components reused e.g. Polymer Web Components
  • Server-Side Components in JavaScript reuse Domain Model
SSJS, CouchDB (Lotus Notes Apps, but SpiderMonkey) Nashorn, Node.js, Yahoo Cocktails, OpenDolphin Concepts

Reactive Applications

Eventstream-driven applications, distributed share-nothing

  • Reactive is Responsive, Elastic, Resilient, Message-Driven
  • Event Bus spanning the Client e.g. Atmosphere, Angular Event Bus
  • Clients retrieve shorter, incremental, update blocks
    e.g. WebSockets, HTML5 SSE, Comet, Long Polling
  • Actor model enables and Event-Sourcing update syncable Cache
  • A centralized Log requires a Domain-Driven-Design

Back-End Implications

Stateless Interfaces, Graceful Degradation, Data Retention

  • Versions forbid heavy state and contracts, e.g.Vert.x
  • Deployments have to be faster, less side-effects
  • State and Sessions need to externalized, security-integrated
  • ...Developers become Polyglot in addition to DevOps
  • Interesting developments in the Java sphere:
    • Java 8 Streams/Async Servlets can be used for constructing
      state e.g. with Kafka
    • JSR 350 State Management
    • JSR 347 Data Grid and Hazelcast, Gemfire
    • JHipster comes with Async support
